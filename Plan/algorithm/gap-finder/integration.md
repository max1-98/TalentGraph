# Gap Finder — Integration

How the Gap Finder connects to the rest of the system: which building blocks
it consumes, how Career Pathing calls it, how results feed User Intelligence,
and what schema extensions are needed.

Depends on: `contracts.md` (trait composition), `algorithm/overview.md`
(building blocks), `career-pathing/overview.md` (step 4), `features/
user-intelligence.md`, `core/schema/api-contracts-algorithms.md`.

---

## Building Block Usage

| Building Block | Usage in Gap Finder | Sub-Traits Consumed |
|----------------|--------------------|--------------------|
| SkillResolver | Indirectly, via RoleContextAnalyser (step 1) | — |
| TaxonomyGraph | Embedding lookup, graph traversal, shortest paths, centrality | SkillProvider + HierarchyReader only |
| RoleContextAnalyser | Step 1: role description to structured requirements | — |
| ProfileComparator | Step 2: candidate tensor + requirements to structured diff | — |
| MarketSignalProvider | Demand, growth, salary data per skill for prioritisation | — |

The Gap Finder never uses GraphMutator or ChangeObserver — it is a read-only
consumer of the taxonomy. This means taxonomy mutations (ingestion, merges)
cannot break gap analysis, and gap analysis cannot corrupt the graph.

## Career Pathing Callsite

Career Pathing invokes Gap Finder at step 4 (`career-pathing/overview.md`)
to convert each candidate destination into a concrete development plan.

**Contract:**
- **Input:** CandidateTensor + a synthesised destination role description
  (generated by DestinationAnalyser from peer trajectory data)
- **Output:** full GapAnalysisResult
- **Extracted fields:** gap count (used as path difficulty), sum of priority
  scores (overall development effort), top bridge suggestions (actionable
  next steps), emd_distance (holistic fit distance)

**Latency target:** less than 20ms per call. Achieved through precomputed
candidate tensors, cached TaxonomyGraph distance matrices, precomputed
centrality metrics, and no LLM calls in the hot path. Career Pathing
evaluates 5–15 destinations per candidate, so per-call latency directly
impacts end-to-end response time.

**Batch optimisation:** when Career Pathing evaluates multiple destinations
for the same candidate, the candidate-side computations (decay-adjusted
proficiency, embedding projections) are computed once and reused. Only the
role-side computations vary per destination.

## User Intelligence Feed

Gap analysis results feed the User Intelligence system (`features/
user-intelligence.md`) through two channels:

**Per-analysis persistence:** each gap analysis result is written to the
gap_analysis_log with candidate ID, timestamp, role context, and full gap
list. This creates a longitudinal record of what gaps the candidate has been
told about.

**Nightly aggregation:** the background aggregator processes the
gap_analysis_log and computes:
- Per-candidate gap frequency: which gaps appear across multiple analyses
- Average priority per recurring gap: identifies consistently high-impact gaps
- These feed into the UserIntelligenceReport.highest_impact_gaps field

**Synergy gap feed:** synergy gaps from SemanticGapDetector (skill pairs with
co-occurrence shortfall) are written to the UserIntelligenceReport.synergy_gaps
field. These power the "You know Python and Kafka separately but never
together" insight described in `features/user-intelligence.md`.

## GapAnalysisResult Schema Extensions

The following fields extend the existing GapAnalysisResult type defined in
`core/schema/api-contracts-algorithms.md`. All new fields are optional to
maintain backward compatibility — existing consumers continue to work
unchanged.

**Per-gap extensions (within each gap entry):**

- **gap_subtype** (enum: missing, weak_slight, weak_significant, decayed,
  experience) — fine-grained classification replacing the coarser gap_type.
  See `proficiency-scoring.md` for classification rules.
- **confidence** (float, 0–1) — how reliable this gap assessment is.
  Propagated from tensor confidence and skill evidence. See
  `proficiency-scoring.md`.
- **pareto_frontier** (boolean) — whether this gap is on the Pareto frontier
  across (Importance, MarketSignal, GraphCentrality). See `prioritisation.md`.
- **semantic_notes** (string, optional) — human-readable annotation when
  semantic detection adjusted the gap: "partially covered by React experience,
  proximity 0.82" or "possibly unstated based on co-occurring skills."
- **proficiency_detail** (object) — structured breakdown: effective
  proficiency, required threshold, and delta. Enables UI-level progress bars
  and detailed explanations.

**Top-level extensions:**

- **emd_distance** (float) — the Earth Mover's Distance between candidate and
  role skill distributions. A holistic fit measure for programmatic consumers.
  See `prioritisation.md`.
- **inferred_skills** (list of objects) — skills the candidate likely has but
  did not list. Each entry includes the skill ID, inferred proficiency, and
  supporting evidence (contributing candidate skills with NPMI scores). See
  `semantic-detection.md`.
- **synergy_gaps** (list of skill pairs) — pairs where the candidate has both
  skills individually but lacks demonstrated co-occurrence. Each entry
  includes the two skill IDs and the co-occurrence shortfall. See
  `semantic-detection.md`.
- **recommended_learning_order** (list of gap IDs) — topologically sorted
  sequence for addressing gaps. See `bridge-suggestions.md`.
